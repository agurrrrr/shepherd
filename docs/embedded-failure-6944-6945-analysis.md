### 2.2 왜 반복 감지기가 발동하지 않았나

`client.go`의 `isDegenerateRepetition`(task #6008에서 도입)은 두 가지 검사를 하는데,
이 패턴은 **둘 다 정확히 회피**한다:

1. **`tailLinesRepeating`** — 마지막 8개 비어있지 않은 줄이 **전부 동일**해야 발동.
   실제 패턴은 `A, B, A, B, ...` 교차라서 인접 줄이 서로 달라 절대 발동 못 함.
2. **`tailPhraseRepeating`** — 반복 단위(period)를 **4~300바이트**까지만 탐색.
   실제 반복 주기는 `A + "\n\n" + B + "\n\n"` = **355바이트**로 상한을 55바이트 초과.
   한글은 글자당 3바이트라 문장 두 개짜리 주기는 거의 항상 300바이트를 넘는다.
   (300 상한은 영어 기준으로 잡힌 값 — **바이트 기반 상한의 한글 취약점**)

그 외 방어선들도 구조적으로 해당 없음:
- `FrequencyPenalty/PresencePenalty 0.3` — 이미 걸려 있었지만 못 막음 (실증됨)
- `isDegenerateOutput` — U+FFFD 비율 검사라서 멀쩡한 한글 반복엔 무반응
- 빈-응답 루프 카운터 — **한 턴 안에서** 계속 생성 중이므로 턴 카운터 자체가 안 돎
- `finish_reason: "length"` 사후 처리 — 발동은 했지만 이미 17분/23K토큰을 태운 뒤

### 2.3 퇴행 자체의 원인

llama.cpp 로그상 `truncated = 0`, 컨텍스트 61,867/94,208 — **컨텍스트 오버플로나
사일런트 context shift가 아니다** (5978/5981 때와 다름). 장시간 에이전트 세션에서
Qwen 계열이 보이는 순수 생성 퇴행(reasoning loop)으로 판단된다. 조기 신호(문단 2회
중복, 중국어 혼입)가 있었고, 컨텍스트에 동일 파일의 페이징 재독(auto-page 경고 포함)이
누적된 상태였다. **퇴행 발생 자체는 로컬 모델에서 완전히 막을 수 없다** — 핵심은
"발생 후 수 초 안에 끊는 것"이고, 그 역할의 감지기가 뚫린 것이 이 실패의 본질이다.

### 3.2 코드상 문제: 모든 스트림 오류 = 즉시 사망

`loop.go:402-409`:

```go
msg, finishReason, usage, err := client.AccumulateStream(ctx, req)
if err != nil {
    return &ExecuteResult{ Incomplete: true,
        IncompleteReason: fmt.Sprintf("API error: %v", err) }, nil
}
```

`unexpected EOF`(연결 절단), `connection refused`(재시작 중), `502 overloaded`(#6943),
idle timeout — 전부 **일시적이고 몇 초~몇십 초 뒤 재시도하면 성공할 오류**인데,
한 번의 실패로 대화 히스토리(`messages` 슬라이스, 이 시점에 메모리에 온전히 존재)를
버리고 작업을 끝낸다. #6945는 11분치 분석 컨텍스트가 통째로 증발했다.

### 3.3 부수 발견들

- **(버그) idle timeout(B1)이 실제로는 스트림을 끊지 못함** — `client.go:107`에서
  `http.NewRequestWithContext(ctx, ...)`로 요청을 만든 **뒤에** `client.go:135`에서
  `ctx, cancel := context.WithCancel(parentCtx)`로 로컬 변수만 재바인딩한다.
  타이머가 발화해 `cancel()`을 불러도 **HTTP 요청에 연결된 컨텍스트가 아니라서**
  `scanner.Scan()`은 계속 블록된다. 주석("close the body to unblock")과 달리 body도
  닫지 않는다. 현재는 "스트림이 다른 이유로 끝난 뒤 에러 메시지만 바꿔치기"하는
  상태다. — 단, **이걸 그냥 고치면 안 된다**: §4.5의 사이드 이펙트 분석 참조.
- **(미상) 8011 취소의 트리거를 현재 로깅으로는 특정 불가** — 8011은 프롬프트 처리
  완료 직후 36토큰 만에 취소됐고 같은 초에 더 큰 요청(8168)이 착수됐다. 트리밍/핸드오프
  경로의 재요청 또는 커넥션 레벨 절단이 후보지만, shepherd가 LLM 요청의 시작/종료/에러를
  전혀 로그하지 않아 확정할 수 없다 (§4.7 로깅 개선 필요). 어느 쪽이든 최종 실패
  원인(수동 재시작)과는 독립적이다.
- **(관측) 토큰 추정에 tools 정의가 빠져 있음** — 요청마다 전송되는 `tools` JSON
  스키마(자체 도구 + MCP 도구, 수천~1만 토큰대 가능)가 `trimMessages`의 추정에
  전혀 계상되지 않는다. 반대로 한글 텍스트는 실제(Qwen 토크나이저 기준 글자당
  ~0.6-0.7토큰)보다 **과대**(글자당 1토큰) 추정된다. 두 오차가 우연히 상쇄되는
  구조라 당장 사고는 없지만, 추정 정확도가 트리밍/핸드오프 타이밍을 좌우하므로
  정리해둘 가치가 있다.
- **(관측) 실패 작업의 토큰 사용량이 DB에 0으로 기록** — #6944/#6945 모두
  `prompt_tokens=0`. 실패 경로에서 usage가 유실돼 사후 분석이 어렵다.

## 4. 개선 방안 (우선순위순, 사이드 이펙트 검토 포함)

### 4.1 【1순위】 일시적 오류 재시도 + 엔드포인트 복구 대기

**문제**: §3.2. #6943/#6945를 모두 직접 해결하고, 향후 서버 재시작·일시 과부하·네트워크
블립에도 작업이 살아남게 하는 단일 개선.

**설계** (`loop.go`의 `AccumulateStream` 호출부):

1. 오류 분류 함수 `isTransientLLMError(err)` 추가:
   - transient: `unexpected EOF`, `connection refused/reset`, `broken pipe`,
     `EOF`, HTTP 5xx(502/503/529 overloaded 포함), idle timeout
   - fatal: HTTP 4xx(400 malformed request, 401/403 인증 등 — 재시도 무의미)
2. transient이면 지수 백오프(예: 2s→5s→15s→30s→60s, 최대 5회)로 **동일 요청 재전송**.
   재시도 전 `GET {baseURL}/models` 헬스체크를 짧은 주기로 폴링해 **서버가 살아날
   때까지 대기**(총 대기 상한 예: 10분 — llama-server 모델 로드가 10~20초 걸리므로
   재시작 시나리오를 흡수). 상한 초과 시 기존과 동일하게 incomplete 처리.
3. 라이브 출력으로 상태 표시: `"⚠️ LLM 서버 연결 끊김 — 재연결 대기 중 (2/5)..."`
   → 사용자가 상황을 인지하고 불필요한 개입을 안 하게 됨.

**사이드 이펙트 검토**:
- 요청 재전송은 **순수 completion 재요청**이라 부작용이 없다 — 실패한 스트림에서
  부분 수신한 내용은 버리고 처음부터 다시 받는다(히스토리 `messages`는 요청 전
  상태 그대로이므로 정합성 문제 없음). 도구는 응답 수신 후에만 실행되므로 중복
  실행 위험도 없다.
- **작업 중지(ctx cancel)와의 상호작용**: 백오프 대기는 반드시
  `select { case <-ctx.Done() ... case <-time.After(...) }`로 구현해 사용자가 작업을
  멈추면 즉시 빠져나와야 한다. `context.Canceled`는 transient로 분류하면 안 됨.
- **핸드오프 경로(`attemptHandoff`)와의 상호작용**: 핸드오프의 스트림 호출은 실패 시
  `ok=false`로 트리밍 폴백하는 기존 동작이 있다. 재시도를 핸드오프 안에도 적용할지는
  선택 사항 — 적용해도 폴백 로직은 그대로 유지되므로 안전하다.
- 무한 재시도 루프 방지: 횟수 + 총 시간 이중 상한. maxIterations 카운트는 재시도에
  올리지 않는다(같은 iteration 내 재시도).

### 4.2 【2순위】 반복 감지기 강화 — 교차(cycle) 반복과 한글 장주기 잡기

**문제**: §2.2. 현행 두 검사를 유지한 채(회귀 방지) 다음을 **추가**한다:

1. **`tailPhraseRepeating`의 주기 상한 300 → 1,200바이트로 상향.**
   - #6944의 355바이트 주기를 포함해, 한글 2~4문장 주기까지 커버.
   - 성능: 검사당 최악 `Σ(p=4..1200) 7p ≈ 5M` 바이트 비교. 400자마다 1회 실행이므로
     23 t/s 로컬 생성 속도에서 완전히 무시 가능한 수준. 그래도 줄이고 싶으면 tail의
     마지막 48바이트를 `strings.LastIndex`로 앞에서 찾아 그 간격을 후보 주기로 삼는
     방식으로 O(window)에 줄일 수 있다 (후보 주기 1개만 검증).
2. **교차 반복 검사 `tailLinesCycling` 신설**: 마지막 12개 비어있지 않은 줄을 모아
   distinct 값이 **2개 이하**이고, 각 distinct 줄이 10자(rune) 이상이며 각각 4회 이상
   등장하면 발동. A/B 교차 패턴을 라인 수준에서 직접 잡는다.

**사이드 이펙트(오탐) 검토**:
- 코드 출력: `})`, `end`, 들여쓰기 등 반복은 "10자 이상" 조건에서 걸러짐. 동일한 긴
  줄이 12줄 중 10줄을 차지하는 정상 출력(예: 표의 동일 행 반복, 로그 덤프 인용)은
  이론상 오탐 가능하나, 이 검사는 **content가 아니라 주로 reasoning_content에서
  발동**하고, 발동 결과는 "스트림 조기 중단 + incomplete"이므로 정상 작업을 완전히
  망치는 급의 리스크는 아니다. 보수적으로 가려면 distinct ≤ 2 조건을 "정확히 2개가
  엄격히 교차(ABAB...)하거나 1개 반복"으로 좁힌다.
- 기존 단위 테스트(`client_test.go`)에 #6944 실제 패턴(한글 A/B 355바이트 주기)을
  회귀 케이스로 추가할 것.

### 4.3 【3순위】 MaxTokens 상한 — 루프 연소 시간 자체를 단축

**문제**: `MaxTokens = ContextTokens/4`는 32K 컨텍스트 시절 설계(=8K). 92K 엔드포인트에선
23K로 부풀어, 감지기가 뚫렸을 때 **한 턴이 17분** 탄다.

**제안**: `MaxTokens = min(ContextTokens/4, 12288)` (또는 엔드포인트 설정
`max_response_tokens`로 오버라이드 가능하게).

**사이드 이펙트 검토**:
- 정상 턴에서 12K 완성 토큰을 넘는 경우는 대형 `write_file`(코드 전체 재작성) 정도.
  이 경우 `finish_reason: "length"`가 뜨는데, 현행 루프는 **content가 비어있지 않으면**
  length여도 그대로 진행한다(도구 인자가 잘리면 `sanitizeToolCallArgs`가 JSON 복구
  시도). 즉 대형 파일 쓰기가 잘릴 확률이 소폭 오른다 — 12288이면 코드 ~2만자 수준이라
  실무상 드물고, edit_file 유도로 회피 가능. 8192보다 12288을 권한 이유.
- 핸드오프 요약(`attemptHandoff`)도 같은 식을 쓰므로 동일 상한 적용 — 요약에 12K면 충분.

### 4.4 【3순위】 length/repetition 실패 시 핸드오프로 부분 결과 회수

**문제**: #6944는 실패 직전까지 26분간 **유효한 분석**(원인 후보 특정)을 해놓고 결과가
0이 됐다. `finish_reason: "length"`+빈 content, `"repetition"` 두 실패 경로 모두
즉시 incomplete로 끝난다.

**제안**: 두 경로에서 즉시 실패 대신 `attemptHandoff`를 1회 시도 → 성공하면 "지금까지의
분석 요약 + 남은 작업 후속 큐잉"으로 **정상 완료** 처리, 실패하면 기존대로 incomplete.

**사이드 이펙트 검토**:
- 퇴행한 것은 **그 턴의 출력**이고, 대화 히스토리에는 퇴행 텍스트가 들어가지 않는다
  (length/repetition 경로는 `messages`에 append 하기 전에 return — 코드로 확인됨).
  따라서 핸드오프 요청은 깨끗한 히스토리로 나간다.
- 단, 같은 모델이 곧바로 또 퇴행할 수 있으므로 **4.2(감지기 강화)가 전제**. 핸드오프
  요청에도 감지기가 걸려 있고(AccumulateStream 공용), 실패 시 폴백이 기존 동작이므로
  최악의 경우에도 현재보다 나빠지지 않는다.
- `ShouldHandoff`(큐 비어있음) 게이트는 이 경로에선 **무시하는 게 맞다** — 트리밍
  대안이 있는 컨텍스트 핸드오프와 달리, 여기선 대안이 "전량 폐기"뿐이기 때문.
  후속 작업이 큐에 추가되는 것은 사용자 관점에서도 실패보다 낫다.

### 4.5 【4순위】 idle timeout 배선 수리 — 반드시 헬스체크와 함께

**문제**: §3.3 첫 항목. `context.WithCancel`을 `http.NewRequestWithContext` **앞**으로
옮기면 한 줄로 고쳐진다. **그러나 그냥 고치면 새 장애를 만든다**:

- llama.cpp는 프롬프트 처리 중 SSE로 **아무것도 보내지 않는다**. #6945에서 실측한
  풀 재처리 시간이 25K토큰=198초였고, 92K 컨텍스트를 꽉 채우면 ~12분까지 침묵이
  가능하다. 현행 5분 타이머가 "실제로 작동하게" 되는 순간, **멀쩡히 처리 중인 긴
  프롬프트를 죽이는 회귀**가 된다 (지금까지 이 회귀가 없었던 건 역설적으로 타이머가
  고장나 있었기 때문).

**제안**: 배선 수리와 동시에, 타이머 발화 시 즉시 abort하지 말고
`GET {baseURL}/models`(2~3초 타임아웃) 헬스체크 → **서버가 응답하면 타이머 리셋하고
계속 대기, 무응답일 때만 abort**. 진짜 죽은 서버(행)만 걸러내고 느린 서버는 기다린다.
이러면 타이머 주기를 5분에서 더 줄여도(예: 2분) 안전하다.

### 4.6 【4순위】 침묵 구간 가시화 — "hang으로 오인 → 수동 재시작" 차단

**문제**: #6945의 방아쇠. 프롬프트 재처리 3~5분 동안 웹UI에 아무 출력이 없어
사용자가 서버를 재시작했다.

**제안**:
- `ChatStream`에서 첫 청크 수신 전까지 30초 간격으로 `OnOutput` 콜백:
  `"⏳ LLM 프롬프트 처리 중... (1m30s 경과)"` — 구현은 idle 타이머와 같은 goroutine에
  얹으면 됨. (첫 청크가 오면 중단; 라이브 출력 전용이라 히스토리 오염 없음)
- 4.1의 재연결 메시지와 함께라면, 사용자가 재시작을 하더라도 **작업은 대기 후
  이어서 진행**되므로 이중 안전망이 된다.

**운영 개선(코드 외)**:
- llama-server에 `--cache-ram` 상향 검토 — 18:45:27의 이빅션(6.5GiB 캐시 한계)이
  풀 재처리 침묵의 시작이었다. 92K 컨텍스트 단일 슬롯이면 캐시 예산을 키우는 것이
  체감 성능에 직결된다.
- 서버 로그의 권고: `chat template supports preserving reasoning, consider enabling
  it via --reasoning-preserve` — Qwen3.6 hybrid 템플릿 사용 중이므로 검토 가치 있음.
- **shepherd 작업이 돌고 있을 때 llama 서비스 재시작은 작업 실패를 유발**한다는 것을
  운영 수칙으로. (4.1이 들어가면 "대기 후 자동 복구"로 완화됨)

### 4.7 【5순위】 관측성 (LLM 요청 로깅 + 실패 시 usage 보존)

- iteration 번호와 함께 요청 시작/완료/에러/재시도를 stdout에 한 줄씩 로그
  (`[embedded] iter=12 req start msgs=45 est_tokens=52000` / `... done finish=tool_calls` /
  `... error=unexpected EOF retry=1/5`). §3.3의 "8011 미스터리" 같은 사건을 다음엔
  로그만으로 규명할 수 있게 한다.
- 실패 경로(`API error`)의 `ExecuteResult`에도 누적 `totalPromptTokens/totalCompletionTokens`를
  담아 DB에 보존 (현재 0으로 기록됨).
- `estimateMessageTokens`에 tools 정의 크기 가산(직렬화 길이 기반 1회 계산 후 캐시).

## 6. 검증 방법

- **재시도**: 테스트용 httptest 서버가 스트림 도중 연결을 끊었다가(1회차)
  2회차에 정상 응답 → 작업이 성공해야 함. `context.Canceled` 주입 시 즉시 종료 확인.
  실제 환경에선 작업 중 `systemctl restart qwen3.6-27b-q6` 실행해 자동 복구 확인.
- **반복 감지**: #6944의 실제 A/B 355바이트 패턴을 `client_test.go`에 고정 케이스로
  추가 — `isDegenerateRepetition` true. 정상 산출물(코드/표 포함 4KB)로 false 확인.
- **idle timeout**: 헬스체크 응답하는 느린 서버(첫 청크 8분 지연) → 살아남아야 함.
  헬스체크도 무응답인 행 서버 → 타이머 발화 후 abort + (4.1에 의해) 재시도 진입.

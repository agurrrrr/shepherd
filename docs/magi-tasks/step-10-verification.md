# Step 10 — 통합 검증 체크리스트 + A/B 측정 준비

> 설계서 참조: §8 (측정), §5.2 (UX), §9 (리스크)
> 선행 단계: step-08, step-09
> **전제**: 사용자가 직접 빌드·설치·서버 재시작을 마친 상태에서 수행한다. **이 단계를 수행하는 에이전트도 바이너리 설치·서버 재시작을 하지 않는다** — 검증 시나리오 실행과 결과 기록만 한다.

## A. 통합 검증 체크리스트

각 항목의 실제 관찰 결과(라이브 출력 원문 일부 포함)를 작업 결과에 기록하라.

### A-1. 설정 경로
- [ ] WebUI Settings > Embedded > MAGI에서 proposer 3개(서로 다른 엔드포인트) + aggregator 설정 후 저장 → `~/.shepherd/embedded.yaml`에 snake_case 키(`endpoint_id`, `confidence_threshold`)로 기록되는지
- [ ] 같은 모델 3개로 구성 시 저장은 되지만 노란 경고 배너가 뜨는지
- [ ] proposer 2개만으로 저장 시도 → 400 + 빨간 에러
- [ ] 저장 후 기존 `endpoints` 목록이 보존되는지 (read-modify-write 검증)

### A-2. 평상 경로 (합의 성공)
- [ ] `magi` 프로바이더 양 생성 → 자문형 태스크 투입 (예: "이 프로젝트의 에러 처리 방식을 평가해줘")
- [ ] 라이브 출력 순서: `🧠 MAGI 심의 개시` → 심의자별 `응답 완료 — 신뢰도 N/10` 3줄 → `✅ 합의 도달` → `📊 MAGI 심의 비용: N 토큰 (호출 4회)`
- [ ] 태스크가 completed로 기록되고 결과에 synthesis가 담기는지

### A-3. 에스컬레이션 경로
- [ ] `confidence_threshold: 10`으로 올려 강제로 토론 진입 → `⚖️ 합의 판정 ... 토론 라운드 진입` → 재판정 → 종결 확인 (호출 8회)
- [ ] 교착 유도가 어려우면: 판정이 계속 합의로 끝나는 것 자체를 기록하고 넘어가라 (교착 렌더링은 단위 테스트로 이미 검증됨)

### A-4. 장애 경로
- [ ] proposer 엔드포인트 1개를 중지시킨 상태에서 실행 → `응답 실패` 1줄 + `2인 심의로 계속` → 정상 완료
- [ ] 2개 중지 → `단일 임베디드 실행으로 폴백` 후 기존 임베디드 경로로 완료
- [ ] `aggregator: claude_cli` 상태에서 (가능하면) 네트워크 차단 → `Claude aggregator 실패 — 로컬 폴백 사용` 확인
- [ ] 태스크 실행 중 WebUI에서 stop → 즉시 중단되고 양이 idle로 복귀하는지

### A-5. 회귀 확인
- [ ] 기존 `embedded` 프로바이더 양의 태스크가 이전과 동일하게 동작 (magi 추가로 인한 회귀 없음)
- [ ] `claude` 프로바이더 태스크 정상

## B. A/B 측정 준비 (설계서 §8 "측정")

`docs/magi-eval/` 디렉토리를 만들고 다음을 작성한다:

1. **`prompts.md`** — 재현 프롬프트 세트. 과거 실패·저품질 태스크에서 발췌:
   - #6944/#6945 부류 (에이전틱 실패 — Phase 1에선 자문형으로 변형해 수록)
   - #7000 부류 (자문형: "플레이 콘솔 접근성 양식에 뭘 적어야 하는지" 같은 컨설팅형 질문)
   - 각 프롬프트에 기대 답변의 채점 기준 3~5줄 (무엇이 있으면 합격인지)
   - 과거 태스크 원문은 `shepherd history` / `get_task_detail`로 조회
2. **`results-template.md`** — 기록 표: | 프롬프트 | 단일 embedded 결과 요약 | MAGI 결과 요약 | 채점 (0-5) | 지연(초) | 토큰 | 판정 경로 (합의/토론/교착) |
3. 실제 A/B 실행은 이 단계 범위 밖 — 사용자가 지시하면 별도 태스크로 수행한다.
